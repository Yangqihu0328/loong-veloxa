# /debug — 系统化调试

你现在以 DEBUG（调试）模式运行。你的职责是系统化地找到问题根因并修复，严格遵循四阶段调试流程。

> **铁律：** 没有根因调查，就不能提出修复方案。

## 执行步骤

### 0. 前置条件检查与状态保存

读取 `memory-bank/activeContext.md`：
- 记录当前阶段为**调试前状态**（用于调试完成后恢复）
- 如果当前有活跃任务 → 将调试关联到该任务
- 如果没有活跃任务（`空闲`状态）→ 在 `tasks.md` 中创建一个独立的调试任务（生成任务 ID `TASK-YYYYMMDD-NN`，复杂度 Level 1，状态设为 `调试中`）
- 将 `activeContext.md` 的阶段更新为 `调试中`，记录调试前状态

### 1. 读取上下文

- 读取 `memory-bank/tasks.md` 获取当前任务信息
- 读取 `memory-bank/activeContext.md` 获取当前状态
- 读取 `memory-bank/techContext.md` 获取技术栈（有助于定位问题）
- 读取 `memory-bank/systemPatterns.md` 获取架构模式（有助于理解系统）
- 了解问题描述

### 1.5 判断调试类型

根据当前阶段和问题性质，选择调试路径：

| 当前阶段 | 问题类型 | 调试路径 |
|---------|---------|---------|
| 初始化/规划中 | 发现现有代码 Bug | 标准四阶段调试（下方步骤 2-5） |
| 初始化/规划中 | 需求矛盾或设计缺陷 | 简化流程：记录问题 → 分析影响 → 提出方案 → 返回 `/plan` 重新规划 |
| 设计中 | 设计方案冲突 | 记录冲突 → 返回 `/creative` 重新探索 |
| 构建中 | 代码 Bug | 标准四阶段调试（下方步骤 2-5） |

### 2. 阶段 1 — 根因调查

遵循 `.cursor/rules/skills/systematic-debugging.mdc` 规则：

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致地重现**
   - 确切的重现步骤是什么？
   - 每次都发生吗？
   - 不可重现 → 收集更多数据

3. **检查最近变更**
   - git diff，最近的提交
   - 新依赖、配置变更
   - 环境差异

4. **追踪数据流**
   - 坏值从哪里来？
   - 什么调用传入了坏值？
   - 向上追踪直到找到源头

### 3. 阶段 2 — 模式分析

1. 在代码库中找到类似的**可工作**代码
2. 与破损代码对比
3. 列出**所有**差异
4. 理解依赖关系

### 4. 阶段 3 — 假设与测试

1. 形成单一假设："我认为 X 是根因，因为 Y"
2. 做最小可能变更测试假设
3. 一次只改一个变量
4. 验证后决定下一步

### 5. 阶段 4 — 实现修复

1. **创建失败测试**（使用 TDD 技能）
   - 先写测试重现 bug
   - 看它失败

2. **实现修复**
   - 修复根因，不是症状
   - 一次一个变更

3. **验证**
   - 测试通过
   - 没有其他测试被破坏
   - 问题确实解决了

4. **3+ 次修复失败？**
   - **停下来质疑架构**
   - 与用户讨论
   - 可能需要重构而非继续修补

### 6. 清理诊断代码

修复完成后，移除阶段 1 中添加的所有诊断日志和临时调试代码。

### 7. 更新 Memory Bank

- 更新 `memory-bank/tasks.md` — 记录 bug 和修复
- 更新 `memory-bank/progress.md` — 记录调试过程、根因和修复方案
- 更新 `memory-bank/activeContext.md` — **恢复到调试前状态**（从步骤 0 记录的调试前状态），清除 `调试前状态` 字段和调试相关临时信息
- **独立调试任务的结束处理：** 如果是从 `空闲` 状态触发的独立调试任务（步骤 0 中创建），调试完成后将该任务在 `tasks.md` 中标记为 `已完成`，并更新任务历史表。不需要走 `/reflect` → `/archive` 流程

### 8. 输出调试报告

```markdown
## 🐛 调试报告

**问题描述：** [问题]
**根因：** [根因分析]
**修复方案：** [实施的修复]
**验证结果：** [测试证据]
**回归测试：** [添加的测试]

**下一步：**
- 从 `/build` 中触发 → 返回 `/build` 继续未完成的任务
- 独立触发（无活跃任务）→ 调试完成，使用 `/van` 开始新任务或结束
- 修复揭示了设计缺陷 → 返回 `/plan` 或 `/creative` 重新设计
```

## 红色信号 — 停下

如果你发现自己在想以下任何一个，**立即停止**并返回阶段 1：

- "先快速修复，之后再调查"
- "试试改 X 看看行不行"
- "同时加多个变更"
- "跳过测试，我手动验证"
- "大概是 X"
- "再试一次修复"（已经试了 2+ 次）

## 关键原则

- **系统化优于猜测** — 15-30 分钟系统化 vs 2-3 小时瞎猜
- **证据先于修复** — 理解问题后再动手
- **一次一个变量** — 不要同时修改多个东西
- **测试证明修复** — 每次修复都要有测试证据
