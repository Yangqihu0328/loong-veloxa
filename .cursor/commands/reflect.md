# /reflect — 回顾反思

你现在以 REFLECT（回顾）模式运行。你的职责是对已完成的实现进行结构化回顾，提取经验教训。

## 执行步骤

### 0. 前置条件检查

读取 `memory-bank/activeContext.md`，验证：
- 当前阶段必须是 `构建中`（由 `/build` 设置）
  - **例外**：若当前阶段形如 `构建中·轮次 N 完成（...）`，说明任务处于多轮次 Build 中间态（见 `.cursor/rules/workflow/complexity-levels.mdc`「多轮次 Build 中间态」），**禁止**进入 `/reflect`，提示用户「任务尚未完成所有 Phase，先 `/build` 续上剩余 Phase」并立即返回
- 如果阶段是 `空闲` 或 `初始化` → 提示用户尚未开始构建，先完成前置步骤
- 如果阶段是 `回顾中` → 提示用户回顾已完成，可以使用 `/archive`
- 验证 `progress.md` 中有实现记录（如果为空，提示数据不足以进行有意义的回顾）

### 1. 读取上下文

- 读取 `memory-bank/tasks.md` 获取任务详情
- 读取 `memory-bank/progress.md` 获取实现过程记录
- 读取实现计划（`docs/plans/` 目录）和创意设计文档（`memory-bank/creative/` 目录，如有）
- 读取设计规格文档（`docs/specs/` 目录，如有）
- 审查 git 历史和实际变更（如果 git 不可用，基于 `progress.md` 记录进行回顾）

### 2.5 回顾检查清单

根据任务类型勾选适用的检查维度（可多选），辅助结构化回顾：

**代码变更类任务：**
- [ ] 计划精确度 — 文件清单是否与实际变更一致？预估是否合理？
- [ ] TDD 执行情况 — 是否遵循了 RED→GREEN？如果使用了变体（覆盖补充/批量验证），原因是什么？
- [ ] 子代理质量（如使用） — 产出是否需要大量返工？prompt 是否足够清晰？
- [ ] 测试隔离 — 是否存在测试间串扰、flaky、环境依赖？
- [ ] 提交粒度 — 是否按计划的子阶段提交？有无大杂烩提交？
- [ ] 非默认路径 — 流式/错误/缓存命中等非主路径是否验证？

**配置/规则类任务：**
- [ ] 文件位置验证 — 是否在修改前 Glob 确认了目标文件位置？
- [ ] 交叉引用 — 修改的规则是否与其他规则文件保持一致？

**安全相关任务：**
- [ ] 输入验证 — 所有外部输入是否经过校验？
- [ ] 认证/授权 — 权限模型是否完整？
- [ ] 数据保护 — 敏感数据是否加密/脱敏？
- [ ] 依赖审计 — 新引入的依赖是否检查过已知漏洞？
- [ ] 错误信息 — 错误响应是否避免泄露内部信息？

### 3. 结构化回顾

按以下维度进行回顾：

#### a. 做得好的

- 什么工作顺利？
- 哪些技术选择效果好？
- 哪些流程步骤特别有价值？

#### b. 遇到的挑战

- 什么出了问题或比预期困难？
- 什么阻碍了进展？
- 哪些假设被证明是错误的？

#### c. 经验教训

- 下次会做什么不同？
- 有什么可以改进的流程？
- 发现了什么新的模式或最佳实践？

#### d. 流程改进

- 头脑风暴阶段是否充分？
- 计划是否足够详细？
- TDD 流程是否被遵守？
- 代码审查是否捕获了问题？

#### e. 技术改进

- 代码质量如何？
- 测试覆盖度如何？
- 有技术债务吗？
- 性能考虑是否充分？

#### f. 安全评估（checklist）

| 维度 | 状态 | 备注 |
|------|------|------|
| 输入验证 | ✅/❌/N/A | |
| 认证/授权 | ✅/❌/N/A | |
| 数据保护（加密/脱敏） | ✅/❌/N/A | |
| 依赖审计 | ✅/❌/N/A | |
| 错误信息脱敏 | ✅/❌/N/A | |
| 敏感数据处理 | ✅/❌/N/A | |

如无安全相关内容，注明"本任务不涉及安全变更"。

### 3.5 反复模式识别

检查本次经验教训是否与以下已知反复模式重复（基于 27 份历史回顾统计）：

| 已知模式 | 出现频率 | 本次是否重复？ |
|---------|---------|-------------|
| 计划文件清单与实际变更不一致 | 9+ 次 | |
| 子代理产出需大量返工（CMake/编译/上下文不足） | 7+ 次 | |
| 前置依赖/环境/API 能力未验证 | 8+ 次 | |
| 非默认路径（流式/错误/缓存）遗漏验证 | 4+ 次 | |
| 测试隔离问题（flaky/并行冲突/环境依赖） | 7+ 次 | |
| 提交粒度偏离计划（大杂烩提交） | 7+ 次 | |
| TDD 严格度与场景不匹配 | 11+ 次 | |

**如果本次教训匹配已知模式：**
- 将该建议优先级**至少升至 P1**
- 在建议中注明"反复出现，需固化到规则"
- 考虑是否需要修改对应的工作流规则文件（`/van`、`/plan`、`/build` 等）

### 4. 保存回顾文档

**Level 1：** 回顾内容直接追加到 `memory-bank/progress.md` 中即可，不强制创建独立回顾文件。
**Level 2-4：** 创建 `memory-bank/reflection/reflection-[task_id].md`：

```markdown
# 回顾：[任务名称]

**日期：** [YYYY-MM-DD]
**任务 ID：** [task_id]
**复杂度级别：** Level [1-4]

## 计划 vs 实际
[对比分析]

## 做得好的
[列表]

## 遇到的挑战
[列表]

## 经验教训
[列表]

## 改进建议（附优先级与落实方式）

| # | 建议 | 优先级 | 落实方式 | 目标 |
|---|------|--------|---------|------|
| 1 | [建议] | P0/P1/P2 | 改规则/加 checklist/记技术债 | [哪个文件或流程] |

优先级定义：
- **P0 立即：** 影响当前工作流正确性，应在本任务归档前落实
- **P1 下次：** 下个同类任务前应落实，迁移到 `activeContext.md` 待处理事项
- **P2 长期：** 记录到 `systemPatterns.md` 或 `techContext.md` 作为长期改进方向

## 技术改进建议
[列表]

## 安全评估
[安全 checklist 结果，或"本任务不涉及安全变更"]
```

### 5. 更新 Memory Bank

- 更新 `memory-bank/tasks.md` — 记录回顾完成
- 更新 `memory-bank/activeContext.md` — 将阶段更新为 `回顾中`
- 更新 `memory-bank/progress.md` — 追加回顾结果
- **反馈到长期知识库**（这一步至关重要，确保跨会话知识积累）：
  - 更新 `memory-bank/systemPatterns.md` — 追加新发现的架构模式和决策
  - 更新 `memory-bank/techContext.md` — 追加技术栈变更、新依赖、技术经验
  - 更新 `memory-bank/productContext.md` — 追加产品功能变化（如适用）
- **改进建议闭环**（防止建议被遗忘）：
  - P0 建议 → 在当前回顾中直接标注落实状态，归档前须完成
  - P1 建议 → 追加到 `memory-bank/activeContext.md` 的「待处理事项」，附来源任务 ID
  - P2 建议 → 记录到 `systemPatterns.md` 或 `techContext.md` 即可

### 5.5 提交回顾产出物

将本阶段所有变更提交到当前分支：
- `memory-bank/reflection/reflection-[task_id].md`（Level 2+ 新建）
- 所有已更新的 Memory Bank 文件（tasks.md, activeContext.md, progress.md, systemPatterns.md, techContext.md, productContext.md）

提交消息格式：`docs(reflect): add reflection for [task_id]`

如果 git 不可用，跳过此步骤。确保提交后 working tree 干净。
