## 遇到的挑战

1. **Color 构造 API 假设错误**：使用 `Color(r,g,b,a)` 构造，实际 Color 是 aggregate struct 须用 `Color::FromRGBA()`。编译错误定位清晰，修复简单
2. **像素格式通道顺序再次混淆**：集成测试中 `r = (px >> 16) & 0xFF`，实际 Veloxa 格式 R 在 bits[0:7]，正确提取应为 `r = px & 0xFF`。这是**第二次**因像素格式导致测试失败（TASK-02 也遇到）
3. **MemorySurface 像素访问**：Application 构造时已 Lock Surface，测试中无法再次 Lock。pixels_ 是 private，无公开访问器。需额外新增 `data()` const 方法
4. **Surface 双重 Lock 约束**：Lock/Unlock 模型的排他性约束未在计划阶段考虑，导致集成测试设计需迭代

## 经验教训

1. **像素格式辅助函数应标准化**：像素通道提取已第二次出错。应创建标准化的 `PixelR()/PixelG()/PixelB()` 辅助函数供所有测试复用
2. **测试基础设施需求应在计划阶段评估**：当测试需要读取内部状态（如 Surface 像素）时，应在计划阶段识别访问路径是否可行，避免实现阶段意外修改 production 代码
3. **aggregate struct 约束**：C++ aggregate 没有用户定义构造函数，只能用 aggregate init `{r,g,b,a}` 或工厂函数 `FromRGBA()`，不能用 `Color(r,g,b,a)`
4. **Lock/Unlock 排他性设计**对测试友好性的影响：data() const 只读访问器是对排他锁模型的合理补充，不破坏互斥语义
