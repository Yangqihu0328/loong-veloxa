**状态：** 已批准（CREATIVE-20260413-01；若与产品预期冲突，可在 `/build` 前书面修订本文件）  
**关联任务：** TASK-20260413-01

## 组件 1：执行预算与中断策略

### 设计挑战

**问题陈述：** 不可信脚本可能死循环或极长运行，占用 CPU；需在嵌入式与 HMI 帧循环之间取得平衡。  
**约束条件：** 首版 `vx_script` 不依赖 `Application` / `EventLoop`；中断回调内不得执行重逻辑或阻塞 I/O。  
**成功标准：** 默认策略可文档化、可测试；进阶策略有明确挂载点，不阻塞首合入。

### 探索的方案

#### 方案 A：首版完全不启用中断

**描述：** 仅依赖实现计划中的源码长度上限与 QuickJS 自带栈限制。  
**优点：** 实现最少，`/build` 最快。  
**缺点：** 无法缓解 CPU 型 DoS（死循环）。  
**复杂度：** 低  
**风险：** 高（车载场景脚本来源一旦开放即暴露）

#### 方案 B：始终开启固定「操作预算」

**描述：** `JS_SetInterruptHandler` + 每 N 次回调递减计数器，每次 `EvalGlobal` 重置预算。  
**优点：** 默认可预测、可单测（注入慢脚本）。  
**缺点：** 合法大脚本可能被误杀；阈值调参依赖业务。  
**复杂度：** 中  
**风险：** 中（误杀导致线上「随机」失败）

#### 方案 C：默认关闭，显式 API 开启 + 与帧循环协作（两阶段）

**描述：**  
- **Phase 1（首合入）：** 不注册 interrupt，与当前 `docs/plans/2026-04-13-quickjs-integration.md` 一致，避免扩大首 PR。  
- **Phase 2（紧随其后的同一 TASK 或子提交）：** 增加 `QuickjsEngine::SetEvalInterruptBudget(usize max_checkpoints)`，`0` 表示关闭；`>0` 时在 `Init` 注册 `JS_SetInterruptHandler`，在 `EvalGlobal` 入口重置计数。回调内**仅**做递减与 `return !=0` 以中止，**不**调用 `EventLoop`。  
- **Phase 3（未来与 Application 集成）：** 由宿主在「每帧」或「定时器」中检查 `QuickjsEngine::WasInterrupted()` 或统一用「协作式取消标志」扩展（不在本文件定 API 细节）。

**优点：** 渐进交付；默认行为与计划一致；车载可在集成层强制开启。  
**缺点：** Phase 1 仍有死循环窗口（以文档与后续里程碑明示）。  
**复杂度：** 中（分阶段）  
**风险：** 低到中（可控）

### 对比

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 复杂度 | 低 | 中 | 中 |
| 可维护性 | 高 | 中 | 高 |
| 性能 | 最优 | 依赖回调频率 | 关闭时零开销 |
| 扩展性 | 差 | 中 | 高 |
| 安全性 | 差 | 中 | 中（Phase 2+） |
| 风险 | 高 | 中 | 低 |

### 选定方案（组件 1）

**方案 C。**  
**理由：** 与已批准的实现计划兼容（首 PR 不增中断复杂度），同时给车载一条明确的硬化路径（Phase 2 + 集成层默认开启）。  
**风险缓解：** 在 `docs/specs` 与归档中写明「未开启 interrupt 前不得对不可信网络脚本开放 Eval」；Phase 2 默认建议值 **10⁷ 级检查点**（具体数值在实现时用常量 + 单测「死循环被中止」校准）。

### 实现指导（组件 1）

- `EvalGlobal` 内**禁止**在中断回调里分配复杂 C++ 对象；仅 `atomic` / `volatile` 计数与返回码。  
- 与 `EventLoop` 的协同**不**在 `vx_script` 内耦合；由未来 `Application` 或宿主线程模型决定。

## 组件 2：`qjs-libc` 与 `js_std_add_helpers` 默认策略

### 设计挑战

**问题陈述：** `quickjs-libc` 提供 `std`/`os` 模块、定时器、信号等能力，显著扩大攻击面。  
**约束条件：** CMake 已链接 `qjs-libc` 以满足静态链接与后续扩展；首版无 DOM、无文件脚本加载。  
**成功标准：** 默认能力面最小；需要 `import std` 时必须有单独安全评审与开关。

### 探索的方案

#### 方案 A：Init 中始终调用 `js_std_init_handlers` + `js_std_add_helpers`

**优点：** 与上游 REPL 行为接近，调试方便。  
**缺点：** 攻击面大（`os`/`std`）；车载常不需要。  
**复杂度：** 低  
**风险：** 高

#### 方案 B：仅链接库，Init 不调用任何 std 辅助函数

**优点：** 能力面最小；`JS_Eval` 纯表达式与函数定义仍可用。  
**缺点：** 日后若需 `console.log` 等需显式加任务。  
**复杂度：** 低  
**风险：** 低

#### 方案 C：编译选项 `QJS_BUILD_LIBC=ON` 合并 libc（改变上游构建）

**优点：** 单库 artifact。  
**缺点：** 与当前计划（独立 `qjs-libc`）不一致；升级 diff 大。  
**复杂度：** 中  
**风险：** 中

### 对比

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 复杂度 | 低 | 低 | 中 |
| 可维护性 | 中 | 高 | 中 |
| 安全性 | 差 | 优 | 中 |
| 扩展性 | 高 | 中（按需加） | 中 |

### 选定方案（组件 2）

**方案 B。**  
**理由：** 与「首版仅 Eval、不信任外部脚本」一致；`qjs-libc` 仍可链接以满足符号与后续可选能力。  
**风险缓解：** 若未来任务需要 `std`/`os`，新增 `ScriptRuntimeOptions::enable_quickjs_std{bool}`，**默认 false**，并配套威胁建模更新。

### 实现指导（组件 2）

- `QuickjsEngine::Init` **不得**调用 `js_std_add_helpers` / `js_std_init_handlers`（除非后续带开关的任务明确加入）。  
- 代码审查清单：搜索误加的 `import` 支持或 `js_std_*` 调用。

## 组件 3：长期内存与分配器对齐

### 设计挑战

**问题陈述：** QuickJS 默认使用系统 `malloc`；Veloxa Foundation 强调 Arena/池化与 OOM 策略。  
**约束条件：** QuickJS 的 `JS_SetGC`/`JS_SetRuntimeOpaque` 与自定义 `JSMallocFunctions` 需长期验证；首版工期有限。  
**成功标准：** 首版有 **硬上限** 防内存型 DoS；自定义分配器有清晰技术债条目，不阻塞合入。

### 探索的方案

#### 方案 A：完全使用 QuickJS 默认堆，不设限

**优点：** 零集成成本。  
**缺点：** 单 Runtime 可被大数组撑爆。  
**风险：** 高

#### 方案 B：`JS_SetMemoryLimit(rt, cap)` + 仍用 libc malloc

**优点：** 几行代码即可限制 QuickJS 堆；与 Foundation 解耦。  
**缺点：** 仍非池化；极限下 QuickJS 行为需读文档与单测。  
**复杂度：** 低  
**风险：** 低

#### 方案 C：立即接入 `JSMallocFunctions` → `vx::MallocAllocator`

**优点：** 长期与 Foundation 统计/策略统一。  
**缺点：** 生命周期、对齐、线程与 QuickJS 多上下文场景需大量设计。  
**复杂度：** 高  
**风险：** 高（首版）

### 对比

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 复杂度 | 低 | 低 | 高 |
| 可维护性 | 高 | 高 | 待评估 |
| 安全性 | 差 | 良 | 良（若做对） |
| 性能 | 中 | 中 | 潜在优 |

### 选定方案（组件 3）

**首版采用方案 B；方案 C 记入技术债（TASK 后续或独立 TASK）。**  
**默认 `JS_SetMemoryLimit`：** 建议 **32 MiB（33554432 字节）** 每 `JSRuntime`，在 `Init` 中设置；提供 `SetMemoryLimitBytes(usize)` 供宿主覆盖（`0` 表示使用默认常量，**不设为无限制** 除非显式负例测试专用构建选项——产品默认应始终有上限）。

**理由：** 在不动分配器vtable的前提下缓解内存 DoS；与嵌入式车载预期一致。  
**风险缓解：** 单测覆盖「分配超大数组失败返回 Status」若 API 可观测；否则文档化依赖 QuickJS 异常路径。

### 实现指导（组件 3）

- 不在本 TASK 引入 `JS_SetGC` 自定义除非与 Phase 2 interrupt 一并验证。  
- `techContext.md` 归档时追加技术债：**自定义 JSMallocFunctions 与 Arena 域对齐**。
